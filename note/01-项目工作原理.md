# OpenAppFilter 项目工作原理

## 一、系统架构

OpenAppFilter 由三层架构组成：

### 1. 内核模块 (oaf.ko) - 深度包检测引擎
- 位置：`oaf/src/`
- 核心文件：`app_filter.c` (1961 行)
- 功能：Netfilter hook 挂载、DPI 解析、特征匹配

### 2. 用户空间守护进程 (oafd) - 配置管理和 RPC 服务
- 位置：`open-app-filter/src/`
- 核心文件：`appfilter_ubus.c` (2483 行)
- 功能：UCI 配置管理、ubus RPC 服务、Netlink 内核通信

### 3. Web UI (LuCI) - 管理界面
- 位置：`luci-app-oaf/`
- 功能：Web 管理界面

## 二、特征库系统

### 特征库格式

```
#version v22.3.24    # 特征库版本 (年.月.日)
#format v3.0         # 格式版本

#class chat 1 聊天    # 分类: 名称 ID 中文名
1003 微博:[tcp;;443;weibo;;]
```

### 应用特征行结构

```
appid name:[proto;sport;dport;host_url;request_url;dict]
```

| 字段 | 位置 | 说明 | 示例 |
|------|------|------|------|
| `appid` | 行首 | 应用 ID = 类别×1000 + 序号 | 3001 (视频类第1个) |
| `name` | 行首 | 应用名称 | 抖音 |
| `proto` | 第1位 | 协议类型 | `tcp` 或 `udp` |
| `sport` | 第2位 | 源端口 | 空表示任意 |
| `dport` | 第3位 | 目标端口 | `443`, `1000-2000` (范围) |
| `host_url` | 第4位 | 主机名匹配 | `douyin` (支持正则) |
| `request_url` | 第5位 | HTTP 请求路径 | `/pull/*.douyincdn.com` |
| `dict` | 第6位 | 十六进制位置特征 | `00:51\|01:48` (位置:值) |

### 多特征支持

同一应用可以有多个特征，用逗号分隔：

```
3001 抖音:[tcp;;;-dy-;;,tcp;;;-dy.;;,tcp;;;douyin;;]
```

```

    ↓
复制特征库到 /etc/appfilter/feature.cfg
    ↓
创建符号链接 /tmp/feature.cfg → /etc/appfilter/feature.cfg

    ↓
```

```c
typedef struct af_feature_node {
```

## 四、数据包处理流程

### 1. Netfilter Hook 入口

数据包首先进入 `app_filter_hook()` 钩子函数，然后调用：

- `app_filter_hook_gateway_handle()` - 网关模式
- `app_filter_hook_bypass_handle()` - 旁路模式

### 2. 协议解析

```c
// app_filter.c:1293
parse_flow_proto(skb, &flow)  // 解析 IPv4/IPv6, TCP/UDP
```

解析结果存储在 `flow_info_t` 结构中：

```c
typedef struct flow_info {
    u_int32_t src, dst;           // 源/目的 IP
    u_int16_t sport, dport;       // 源/目的端口
    int l4_protocol;               // L4 协议 (TCP/UDP)
    unsigned char *l4_data;        // L4 数据指针
    int l4_len;                    // L4 数据长度
    http_proto_t http;             // HTTP 解析结果
    https_proto_t https;           // HTTPS 解析结果
    u_int32_t app_id;              // 匹配的应用 ID
} flow_info_t;
```

### 3. DPI (深度包检测)

```c
// app_filter.c:1207-1214
int dpi_main(struct sk_buff *skb, flow_info_t *flow) {
    dpi_http_proto(flow);    // 解析 HTTP
    dpi_https_proto(flow);   // 解析 HTTPS (SNI)
    return 0;
}
```

#### HTTP 协议解析

- 提取 HTTP 方法 (GET/POST)
- 提取 Host 头
- 提取请求 URL

#### HTTPS 协议解析

- 识别 TLS ClientHello 报文
- 提取 SNI (Server Name Indication) 扩展
- SNI 位置偏移：`HTTPS_URL_OFFSET = 9`

## 五、完整数据流

```
特征配置文件 (feature.cfg)
    ↓
用户空间: init_app_name_table() → app_name_table[]
    ↓
用户空间: af_load_feature_to_kernel() → Netlink 发送 (ID 29)
    ↓
内核空间: af_init_feature() → add_app_feature()
    ↓
内核空间: 存储到 af_feature_head 链表
    ↓
数据包到达 Netfilter Hook
    ↓
parse_flow_proto() - 解析 L3/L4
    ↓
dpi_main() - HTTP/HTTPS 解析
    ↓
match_feature() - 遍历特征链表匹配
    ↓
af_match_one() - 协议 → 端口 → URL → 位置特征
    ↓
匹配成功 → 记录 app_id → 更新 conntrack mark
    ↓
match_app_filter_rule() - 检查用户规则
    ↓
执行放行或丢弃操作
    ↓
更新客户端访问统计信息
```

## 六、通信机制

### 内核与用户空间通信

- **Netlink 协议**：ID 29
- **消息类型**：
  - `AF_MSG_INIT` - 初始化
  - `AF_MSG_ADD_FEATURE` - 添加特征
  - `AF_MSG_CLEAN_FEATURE` - 清理特征

### 用户空间 RPC

- **ubus 服务名**：`appfilter`
- **主要方法**：
  - `dev_list` - 列出所有设备
  - `visit_list` - 设备访问历史
  - `get_app_filter` / `set_app_filter` - 应用过滤配置
  - `get_app_filter_time` / `set_app_filter_time` - 时间规则

### 配置存储

- **UCI 配置**：`/etc/config/appfilter`
- **特征库**：`/etc/appfilter/feature.cfg` → `/tmp/feature.cfg`
- **日志**：`/tmp/log/appfilter.log`
